计算机网络

计算机网络

简介

计算机网络是什么？

计算机网络是由通信线路相互连接而组成的网络系统

计算机网络是由管理、应用、硬件共同构成的

数据通信

数据通信是三个学科思想和方法的结合

  • 物理学
  • 电气工程
  • 数学

数据通信的动机和范围

数据通信具有三个主要动机

  • 信息源可以是任意类型

    主要考虑到多媒体技术的发展，信息可以是多种形式的

  • 传输系统采用物理系统

    主要是用自然现象进行信息的传递

  • 多个信息源共享底层的传输介质

    共享是根本

数据通信的简单观点

  • 二进制数据
  • 无差错
  • 高性能

传输介质

传输介质的种类

按照路径类型进行分类

可以分成导向以及非导向传输

导向传输即通信可以沿着确切的路径传输信号，非导向传输则相反

导向传输：

  • 安全性
  • 高性能

非导向传输：

  • 便携
  • 减少线路架设的成本

按照能量形式进行分类

背景辐射和电器噪声

计算机网络不能使用正常的电网，因为一下几个原因

  • 随机电磁辐射弥漫于环境之中
  • 电磁辐射碰到金属的时候会产生微弱的信号，这种信号会干扰通信
  • 介质之间放置足够的金属可以起到屏蔽作用

导向介质

铜线

  • 便宜
  • 容易安装
  • 高性能

但是存在电气干扰

双绞线

双绞线可以分为

  1. 屏蔽双绞线
  2. 非屏蔽双绞线

同轴电缆

外层有一个金属网的屏蔽层，可以从任何方向阻挡电磁辐射

但是这种方式有一种缺陷：柔韧性非常差

光纤

一般使用玻璃或者透明塑料

  • 无干扰
  • 传播距离远
  • 编码更多信息
  • 单个

但是光纤也有很多缺点

  • 很难安装
  • 容易破损
  • 很难修复

光纤具有很多种类

  • 多模突变：最便宜，边界密度突变，频繁反射，色散高
  • 多模渐变：边界密度渐变，减少反射，降低色散
  • 单模：最贵的，色散最少

红外

优点

  • 很便宜
  • 容易安装

缺点

  • 范围非常小
  • 容易被阻断（即便是水蒸气）

点对点激光通信

  • 长距离
  • 容易被阻挡
  • 不能穿过植被

电磁波

长距离传输且能够穿过障碍物，便宜，便携

但是很不安全且限制于频率

分类 频率范围 传播方式
低频段 <2MHz 沿地球表面传播，会被地形阻挡
中频段 2~30MHz 可以被大气层或者电离层反射
高频段 >30MHz 按直线传播并且会被障碍物阻断

卫星

使用电磁波进行通信，传输长距离，非常昂贵

轨道类型 描述
低地球轨道 低时延，但是位置不断移动
中地球轨道 轨道为椭圆形，一般提供南北两极之间的通信
地球静止轨道 固定在相对地球的某一位置，但是距离远，时延比较长

近地轨道的高度一般为35785km

最少使用三颗近地轨道卫星可以覆盖整个地球

对于传输介质的度量

  • 传播时延：信号往返一次所需要的时间
  • 信道容量：介质可以支持的最大数据速率
  • 带宽：硬件能够改变信号📶的最大程度（Hz）
  • 数据速率：每秒传输的最大字节数

奈奎斯特定理

D是最大数据速率，B是带宽，K是使用的信号电平的数量

奈奎斯特定理是在没有信号干扰的情况下描述最大数据速率和带宽之间的关系

噪声对于通信的影响

奈奎斯特定理是一个不能达到的理论上的最大值

因为实际上会受到噪声的背景辐射的干扰

香农定理

C为对于信道容量的有效限制，B为带宽，S/N是信噪比

信噪比使用分贝(dB)进行度量

信道容量的意义

  • 奈奎斯特定理鼓励工程人员探索比特信号的编码方法，因为巧妙的编码方式可以在单位时间传输更多的数据
  • 香农定理告诉工程人员：不存在任何巧妙的方法能够图谱物理定律的每秒最大可传输比特率的根本限制

调制与调制解调器

传输模式的分类

传输模式主要有两个基本类型

并行传输

并行传输是：在分离的媒体中同时传输具有多个数据位的传输机制

并行传输一般具有多根独立导线的有线介质，其中每一根导线之间之间必须同步

并行传输有两个最主要的优点：

  • 高吞吐量：可以并行同时发送N比特
  • 与底层硬件相匹配：在计算机内部使用的也是并行电路，两者逻辑相符

串行传输

串行传输一次只发送一个比特

绝大多数系统都适用串行传输

使用串行传输的原因：

  1. 串行传输的成本低，需要很少的导线
  2. 并行传输需要每一根导线要完全一样长，要求过于严格
  3. 高速传输的时候并行传输的导线之间会产生电磁噪声，干扰其他信号

但是使用串行传输需要将硬件的并行数据转换为串行数据（一般使用同步异步收发器）

传输顺序

  • 大端：MSB即先传输最高位
  • 小端：LSB即先传输最低位

选择其中任意一种方式都可以，但是发送与接收必须使用同一种方式

串行传输的定时

串行传输主要有三种定时方式

  1. 异步传输：可以从任意时间开始，时间间隔也是任意的
  2. 同步传输：数据项连续不断传输，数据项之间没有间隔
  3. 等时传输：数据项在规定的时间区间上进行传输，数据项之间的间隔是固定的

异步传输

异步传输有一个问题：发送器和接收器之间缺少协调——当介质空闲的时候，接收器不知道数据要来之前还要空闲多久，因此在发送数据的时候会在前面添加冗余位以提醒接收器接收数据

如在RS232中就会使用一个额外的0比特来表示数据传输的开始，而在传输之后使用一个额外的1比特表示数据传输的结束

同步传输

同步传输最大的优点在于同步开销非常小，基本上不需要同步，不需要开始位和结束位

等时传输

是一种同步传输的重要方式，主要用于传输音乐以及视频等多媒体应用

通信信道

通信信道主要有以下三种类型

通信信道 描述
单工信道 只能单方向传输信号，类似于广播或者电视
全双工信道 允许两个方向的信号同时传输，类似于电话的情景
半双工信道 能够进行两个方向的通信，但是同一时间内只能尽心给一个方向的通信，类似于对讲机

DCE设备和DTE设备

  • DCE即数据通信设备，即用于通信的设备
  • DTE即数据终端数据，即最终接收到信息的设备

调制解调器一般在DCE的位置

调制与调制解调器

基本概念

载波：远距离通信系统都使用连续震荡的电磁波

载波主要有以下属性：

  • 能够传播的比其他的信号更远
  • 能够通过多种媒介进行传输
  • 是连续的
  • 构建了长距离通信的基础

调制：根据正在发送的信息进行对于载波的改变

解调：将载波重新变回数据

模拟调制方案

根据载波的方程

其中A为振幅，f为频率，p为相位

根据这三个参数可以有三种调制方案

  1. 振幅调制
  2. 频率调制
  3. 相位调制：理论上可行但是实际上很少使用

振幅调制

振幅调制一般只是轻微发生变化

因为根据香农定理：

信噪比越大则最终的信道容量越大，而振幅与信号的能量相关，因此需要尽可能保持较大的振幅，因此振幅的变化一般不会很大

最终信号存储在载波的包络曲线中

频率调制

频率调制时振幅保持不变，频率发生变化，信号增强的时候频率提高，信号降低的时候，频率下降

相位调制

对于相移进行控制理论上时可行的，但是却很少用于模拟信号，这种调制方式主要用在数字信号的调制上

对于模拟信号，相位调制类似于频率调制，可以认为是一种特殊的频率调制

键控

对于模拟信号进行的一般称为调制，但是对于数字信号进行的改变一般被称为键控

键控不采用连续的任意值进行调制，只使用固定值的集合

  • 幅移键控：通过制定两个固定值的高电平/低电平表示1/0

  • 频移键控：使用两组固定频率表示0/1

    • 幅移键控和频移键控发送一个字节至少需要一个完整的波

  • 相移键控：使用突然改变载波波形的相位进行数字的编码

    • 相移键控一般使用星座图进行编码

    • 相移键控能够每次变化表示多余一个数据位(因为三角函数是可以分解的)

调制解调器

调制器：将比特信息调制到载波波形上的机器

解调器：将载波上存储的信息解调成比特信息的机器

一般调制器和解调器会被做到一个机器上（如上）

调制解调器主要有三种类型

  • 射频解调器：电磁频率
  • 光纤解调器：光纤
  • 拨号解调器：使用电话系统进行通信
    • 使用音调作为载波而不是电子信号
    • 这个过程非常复杂，因此速率一般比较低：计算机内是数字信号，经过调制器变成声音信号，传到电话局后又变成数字信号传导另一个电话局。这个过程非常繁琐
    • 数据带宽比声音的带宽小

复用与解复用

  • 复用：来自多个信源的信息流组合在一条共享介质中传输，并用复用器实现

  • 解复用：将组合信息分离出各自的信息流

复用主要有以下类型：

  1. 频分复用
  2. 波分复用
  3. 时分复用
  4. 码分复用

频分复用

使用的物理学原理：

一组无线电台如果各自使用不同的频道，就可以同时发送电磁信号而不会相互干扰

发送的时候只需要将不同频率的信号直接混合发送即可，而解复用则需要使用一组过滤器，每一个过滤器过滤特定的频率范围即可

这其中使用的关键思想：过滤器仅仅检查频率

FDM最大的优点在于：多对通信实体能够同时使用一条传输介质

防护带

但是还应该对于信道的频率范围进行一定的限制

  • 因为如果频率相近，则会出现信号干扰

因此频率之间采用一定的间隔，称为防护带

频率范围

即使载波使用单一频率，但是还是应该分配一个频率范围

频分复用的主要性质

  1. 历史悠久：从早期无线电通信实验开始就是这么做的
  2. 广泛使用
  3. 模拟：中间传输的是模拟信号
  4. 通用：只过滤频率不检查其他属性

而由于频分复用的模拟性质，导致对于噪声和失真非常敏感，但是同时也有很好的灵活性

因此系统使用频段主要有两种方式：

  • 增加数据速率
    • 把信道的频率范围划分为多个载波，每一个载波上发送数据（实际上是子信道分配）
  • 增加抗干扰能力
    • 使用扩频的技术，将信道划分为多个载波，每一个载波发送相同数据，接收器选择差错最少的副本

分级FDM

每一级的输入来自于上一级的输出

波分复用

波分复用就是光纤中的频分复用

原理为：

波长不同的光组成的光线能够在棱镜的作用下分解为不同波长的光（即光的分解与合成）

时分复用

时分复用即不同信号源在同一介质上在不同时间发送信息

同步时分复用

时分复用存在多种形式，最经典的是同步时分复用，即不同信号源轮流在信道上发送信号

但是由于系统不停发送信息，而数据项出现的时候没有任何标示，因此解复用器无法判断数据项从哪里开始，如果时钟出现一点偏差，就会导致解复用器无法解析信息

因此在电话系统中，会添加一个成帧信道，每一循环插入一个比特，如果发生差错导致解复用器丢失比特，就很容易能够检测到这个差错并开始重传

分级TDM

与频分复用类似，TDM也可以是层次结构

但是区别是：

  • 时分复用每一个后一级有N倍比特速率
  • 频分复用每一个后一级有N倍频率

同步TDM的问题

实际上使用时分复用的时候，由于数据不是连续产生的，而总是突发产生的，因此在突发产生的时候会产生一定的空转时间。

但是在实际应用中，底层系统必须连续传输数据，因此空转时间会被硬性填充为0，并设置额外比特指定该值是无效的

统计时分复用

统计时分复用可以解决该问题：
以轮流的形式进行数据项传输，跳过未准备发送的数据，而不保留未填充的数据，这样可以使用更短的时间发送等量的数据，但是解复用的时候需要额外开销

逆复用

将多条低速的传输介质逆复用产生一条高速传输介质（但是需要解决不同线路上的时延的问题）

码分复用

码分复用常被应用在电话系统和卫星通信系统中，其不依赖物理特性，而是通过数学方法

正交向量空间中的值可以互不干扰的复合和分离

在码分复用中，每一个发送器都有一个独特的码片序列

如

发送器 码片序列 数据值
A 1 0 1 0 1 0
B 1 1 0 1 1 0

(一般把0看成-1)

那么A的码片序列(1，-1)，数据值(1，-1，1，-1)

B的码片序列(1，1),数据值（-1，1，-1，-1）

注意两个发送器的码片序列应该是正交的

用A的码片序列乘数据值得到((1，-1),(-1，1),(1，-1),(-1，1))即(1，-1，-1，1，1，-1，-1，1)

同理得到B的(-1，-1，1，1，1，1，-1，-1)

将两个复合得到(0，-2，0，2，2，0，-2，0)将该串通过信道进行传输

如果需要解析出A的信息只需要用A的码片序列乘这个传输值即可

CDM最主要的优点是伸缩能力，能够很好的在高利用率网络中保持低时延

而不像TDM需要等待到其发送数据的时候才能够发送数据

解决中文问题

问题

今天在调整软工项目的时候，忽然发现我的微信小程序并不支持中文，很难受

修改数据库字符设置

我一开始以为只是我的数据库的问题，然后我使用了一下指令，将Mysql数据库的全局变量全部设定为了UTF-8

结果我欣喜地发现，我的数据库可以进行中文的数据的输入了，然而我尝试从小程序处通过http请求进行中文内容的输入，但是结果发现在发送请求之后，数据库中本来中文的地方全部变成了问号。不行，还是要继续修改啊

修改微信小程序请求头

之后继续查资料，发现还可能是微信小程序发送http请求的时候需要在请求头的地方添加字符的编码形式

但是这样修改之后我发现还是不能进行正常的中文发送请求，还应该继续修改

修改Springboot配置url

最后我查到还是要修改Springboot的url配置，前后端以及数据库的全部字符集设置为utf-8才能准确进行中文的解析

注意如果需要支持中文一定要添加？后面的一长串东西

总结

如果需要支持中文需要将前端，数据库，后端这里的每一个环节设定为统一的字符集

  1. 将数据库的字符集进行统一设定
  2. 设置前端的请求头，表示请求体以这种字符集的方式进行发送
  3. 设置后端的数据库连接url

完成以上四步应该就能正常使用中文

API

前言

在实际上的接口设计中，有很多中接口的设计风格，其中最主要的有以下三种

  1. RPC
  2. RestFul
  3. GraphQL

在实际使用的过程中应该选择一种比较合适的方式进行接口API的设计

RPC风格

RPC风格是本地调用远程的方式，同时也是最常用的方式

  • RPC的组织形式是类和方法
  • API的名称大多数情况下是动词
  • 一般在这种情况下url会有比较复杂的形势，因此需要一份比较详细的接口文档
  • HTTP方法一般只使用GET/POST

在实际生产中有50%以上的使用这种风格的接口

Rest风格

简介

是一种架构风格，有四种成熟度

  1. 定义一个url，所有操作是对此url的POST操作
  2. 为各个资源创建单独的url
  3. 使用HTTP方法定义对于资源执行的操作的类型
  4. 使用超媒体

其中3是真正的REST风格

REST的所有API都是围绕资源的操作进行的，与RPC不同一般使用不同的HTTP请求类型来区分对于同一资源的不同操作

而RPC则是实用不同的名称进行不同操作的区分

  • GET：获取资源列表 如user/1即获取id为1的用户信息列表
  • POST：创建资源，为集合添加新的资源
  • PUT：全部修改资源，用于一整个资源的更新
  • PATCH：部分修改资源：请求体是修改部分的内容
  • DELETE：移除资源，与GET相类似，如user/1就是移除id为1的用户
APIName 非RESTFUL RESTFUL
获取用户 /user/getUser /users/1 GET
删除用户 /user/deleteUser /users DELETE
新增用户 /user/addUser /users POST
更新用户 /user/updateUser /users PUT

存在的问题

操作繁琐，意义不大

使用 HTTP 的 GET\POST\PUT\DELETE 来区分操作资源，HTTP Method 本身就对外部不友好，是隐藏的方法，把动词加入到 url 中，反而清晰可见，简单易懂，为什么一定要用 url 来表示资源而不能加动词呢。

GET\POST\PUT\DELETE 的兼容性有待认证，首先是兼容老的系统，大部分 HTTP 应用是基于 GET/POST 来实现的。

过分强调单一资源

这可能是人的理解问题，就看我们对资源的定义，太多的场景，如果使用 Restful API 规则行事，势必要把一个 API 拆分多个 API，框架多个 API 之间的状态又成了问题。

这种方式对于处理多资源共同交互的时候会非常麻烦

返回值问题

使用 HTTP Status 状态码是没有问题的，但使用不常见的状态码表示操作层面的含义，对开发不是很友好。
对返回集中定义 status、message 例如下面的 json 返回是很常见的返回，简单易懂。并没有什么不妥，但在 RESTful Api 拥护者眼里就是错误的格式，错误的返回。

更高的成本

统一为 RESTful API 风格，强化要求做的完美，比如带来更多的时间成本、人力成本。沟通成本。

GraphQL风格

这种风格的API由于实用的比较少，而且主要是用在图数据库中因此不在这里详细介绍

参考知乎

总结

在下列情况考虑RPC风格的API或说是RPC：

  • 偏向内部的API
  • 没有太多的时间考虑API的设计或没有架构师
  • 提供的API很难进行资源、对象抽象
  • 对性能有高要求

在下列情况考虑REST风格：

  • 偏向外部API
  • 提供的API天生围绕资源、对象、管理展开
  • 不能耦合客户端实现
  • 资源的CRUD是可以对齐的（功能完整的）

在下列情况考虑GraphQL：

  • 客户端对于数据的需求多变
  • 数据具有图的特点

The align-items property applies to all flex containers, and sets the

HTTP

什么是HTTP

HTTP是一个简单的响应请求协议，通常运行在TCP之上

  • 文本： html，字符串
  • 超文本：图片，音乐，视频
  • 一般使用80端口

两个时代

  • http1.0
    • HTTP1.0:客户端可以和web服务器连接后，只能获得一个web资源，断开连接
  • http2.0
    • HTTP1.1:客户端可以和web服务器连接后，可以获得多个web资源

HTTP请求和响应

请求

客户端—发请求（request）—服务器

请求行

请求方式：

  • Get

    一次请求能够携带的参数比较少，大小有限制，会在浏览器中的url中显示，不安全，但是高效

  • Post

    请求能够携带的参数没有限制，大小没有限制，不会在浏览器中显示数据内容，安全，但是低效

请求头

响应

服务器—响应—客户端

响应体

响应状态码

  • 2xx 响应成功 一般为200
  • 3xx 请求重定向。一般为304
    • 重定向到其他url中
  • 4xx 没有找到对应资源 一般为404
  • 5xx 服务器代码错误 一般为500 （502为网关错误）

题目

乌龟棋的棋盘是一行NN个格子，每个格子上一个分数（非负整数）。棋盘第1格是唯一的起点，第NN格是终点，游戏要求玩家控制一个乌龟棋子从起点出发走到终点。乌龟棋中MM张爬行卡片，分成4种不同的类型（MM张卡片中不一定包含所有44种类型的卡片，见样例），每种类型的卡片上分别标有1,2,3,41,2,3,4四个数字之一，表示使用这种卡片后，乌龟棋子将向前爬行相应的格子数。游戏中，玩家每次需要从所有的爬行卡片中选择一张之前没有使用过的爬行卡片，控制乌龟棋子前进相应的格子数，每张卡片只能使用一次。游戏中，乌龟棋子自动获得起点格子的分数，并且在后续的爬行中每到达一个格子，就得到该格子相应的分数。玩家最终游戏得分就是乌龟棋子从起点到终点过程中到过的所有格子的分数总和。很明显，用不同的爬行卡片使用顺序会使得最终游戏的得分不同，小明想要找到一种卡片使用顺序使得最终游戏得分最多。现在，告诉你棋盘上每个格子的分数和所有的爬行卡片，你能告诉小明，他最多能得到多少分吗？
