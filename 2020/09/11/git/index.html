版本控制

版本控制是一种记录一个或者若干个文件变化，查看文件版本的工具。不仅能够对于源代码进行版本控制，还能够对几乎任何文件进行版本控制

本地版本控制

在最开始的时候，一般使用本地版本控制

即直接在一个本地数据库中记录一个文件的不同版本信息

但是使用这种方式有一个很大的问题，即不能够实现多个贡献者共同向版本控制器中提交

集中化版本控制

为了解决多个贡献者同时工作的问题，一般采用以下的解决方案

即将所有的文件版本存储在一个远程服务器中，所有贡献者都可以从该服务器中获取文件。同时这种方式也比每一个人都采用本地版本控制更容易进行管理。

但是这种方式有两个非常严重的问题，就是如果服务器宕机了一段时间，那么在这一段时间中，所有的贡献者都无法进行工作的提交，同时如果服务器出现了故障，那么所有的数据备份都会消失，是非常不利于工作的

分布式版本控制

为了解决以上的问题，采用了分布式版本控制。即在每一个客户端都对仓库进行一个镜像的备份，这样如果主机出现了问题也能进行工作。只需要在主机修理完成后，再进行提交即可，结构如下

git就是该类型的一种

git基础

核心思想

git与其他版本控制系统的最大差别就是

git采用的是快照流的形式而不是差异比较的形式

也就是说git记录的是每一个版本中的文件内容，而不是像其他版本控制系统的只保存文件与之前的差异，即快照流

快照流形式：

git一般只向其中添加数据

一般只会向git中添加数据，因为git不赞成进行不可逆的操作（因为添加文件是可逆的）

因此一旦将数据提交到git中就很难再丢失数据

三种状态

git中的文件有三种状态

  1. 已修改：对文件进行了修改且还没有提交到数据库中
  2. 已提交：已经提交到数据库中
  3. 已暂存：文件的当前版本已经存到下次提交的快照之中

因此刚好对应git的三个工作区域

  1. 工作目录：从git仓库中获取的，一般放在磁盘上供修改的地方
  2. git仓库：git保存项目数据及文件的地方
  3. 暂存区：保存下次需要提交的文件列表

git工作流程：

先在工作区域进行修改，创建新文件等操作，再将工作区域的内容提交到暂存区域，之后将暂存区域的文件提交到git仓库中

查看提交历史

使用git log可以查看仓库的提交历史

会显示每一次提交的hash值，同时显示每一次提交的描述

撤销并重新提交

有的时候在提交完成之后，会发现提交的代码存在问题，或者提交的时候遗漏了一些文件，需要撤销该次的提交

这种方式提交会覆盖上一次的提交，此时上一次的提交不会留下任何记录而被替换成了这次的提交

如果发现一个提交的文件存在问题，此时可以先在工作区域进行修改，然后再使用git add 被修改的文件,然后使用git commit amend，就可以用这次的提交覆盖上一次错误的提交

注意这种方式和不带参数的git commit的最大的差别是带参数的版本不会留下上一次错误提交的记录，而可以视为对上一次提交大了一个补丁

取消暂存文件

可以使用git reset HEAD filename来取消暂存区中的该文件，这样在下次提交的时候就不会提交该文件

撤销修改

有的时候对自己的工作不满意，觉得文件已经被改的混乱，需要重新进行修改，此时可以使用git checkout -- filename，将工作区域中的该文件的全部抹出，恢复到上一次提交的状态

这个操作的实质就是将仓库中的版本的该文件导出，覆盖掉工作区域中的该文件

从远程仓库抓取

如果本地仓库是使用git clone创建的，可以使用git fetch抓取上一次克隆之后推送的全部的工作

注意fetch是不会进行合并的，所以需要在条件成熟的时候进行手动合并

如果需要进行自动合并可以使用git pull

这个指令会从当初clone的服务器中抓取数据，然后自动尝试合并到当前分支处

推送到远程仓库

如果需要将本地仓库中的全部推送到远程仓库中（注意一定是经过了commit的文件即仓库中的才会被推送）可以使用指令git push [remote_name] [branch_name]

即可以指定push到指定的远程服务器中的远程分支中

如果有两个工作者都是从同一个节点克隆的文件，然后分别进行工作，如果工作者1先提交，那么工作者2就不能使用上面的命令，而是需要再将工作者1提交之后的内容克隆下来，再其基础上更改，最后才能push到远程仓库中

git分支

分支简介

git保存的是文件快照而不是文件的差异

每一次提交的时候，git都会创建一个提交对象（提交对象中记录了提交的基本信息，如作者，提交者之类的），这个提交对象会指向一个树对象，而树对象中记录了提交的文件结构且指向其中的每一个文件，每一个文件节点就是当前的文件快照

而如果对提交做出了修改，那么修改之后的提交对象会有一个父指针指向被修改的提交

而分支，就是指向提交对象的可变指针

如上图，红色的部分就是指向提交对象的指针即分支，其指向的就是该分支当前的提交，而黄色的HEAD指针指向的就是当前所在的分支（即上图表示当前所在分支为master分支）

创建分支

如果需要创建分支，可以直接使用指令git branch [branchname]（创建时指向的提交默认为与当前所在分支相同）

这样两个分支都指向了哈希值为$f30ab$的提交，而HEAD指向master表示当前分支指向的是master分支

如果需要切换分支，就可以使用指令git checkout [branchname]

特别注意，使用checkout指令切换分支的时候，会更换使用者在工作区域的文件，即在切换分支的同时将工作区域中的文件切换为你切换后分支对应的文件
