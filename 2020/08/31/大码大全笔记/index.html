大码大全笔记
2020-09-06|技术总结

ADT

  1. 在对一个属性进行修改的时候，最好使用方法进行修改，而不是直接使用Attribute=xxx这种形式，这样可以隐藏实现细节且更具有可读性setAttributeXXX()

  2. 尽量将底层数据结构封装为ADT，如堆栈，列表等，在实际使用的时候，不要讲存储了员工的列表而是直接认为是一些员工，采用更高的抽象级别。

  3. 将文件这样的常见对象都封装成ADT，将主要的文件操作都封装起来放在ADT中充当方法，可节省大量的重复劳动
  4. 如果一个类中的某些操作之间没有明确的关联，最好是将其分散到不同的类中，不要混杂起来

良好的类接口

  1. 类的接口应当有统一的抽象层次

    比较好的抽象方法

    如果不使用好的抽象方法，程序会越来越难以理解

  2. 注意程序应该提供成对的操作，比如有灯的开，就有灯的关闭，在写类的时候，一定要检查操作的对应操作是否存在

良好的封装

  1. 尽可能地限制类和成员的可访问性

  2. 不要公开暴露出成员数据，尽可能使用访问器来控制访问

  3. 不要对于类的使用者作任何假设，如在程序中注释

    不应该对接口被如何使用作任何假设，而应该在实现的过程中考虑到这些情况出现时的处理

  4. 在cpp中尽量避免使用友元类

  5. 阅读代码的次数会比写代码的次数多得多，因此应该时刻保证代码的可读性

  6. 留意不同类之间的耦合关系，不同类之间的耦合程度应该越低越好

类的设计与实现

包含关系（has a）

包含关系一般使用类的成员进行建立这种关系

如一个雇员有一个姓名，一个电话，这是就可以将姓名，电话作为雇员的成员

注意，如果一个类具有超出[5，9]的范围的数据成员时应当警惕是否能够拆分为多个类，如果是简单数据类型的成员按上限考虑，如果是复杂对象的话按下先考虑

继承关系（is a）

继承是指一个类是另一个类的特化，目的是定义一个包含多个类共有元素的基类以写出更精简的代码

  1. Liskov替换原则

    除非一个派生类真的是一个更特殊的基类，否则不应该从基类继承，换句话说一个基类中定义的方法，必须在所有派生类中的含义是相同的，至少是近似的

    因此这两个类并不能从Account类中继承，因为InterstRate在两个子类中意义有明显差异

    假设一个类CheckingAccount中有InterestRate()是返回银行付给消费者的利息

    而 LoanAccount中的InterestRate()是消费者向银行支付的利息，

    那么假设调用者是银行，就需要时刻记着这两个的符号是相反的，因此显然这样会造成语义混乱

​ 因此这两个类并不能从Account类中继承，因为InterstRate在两个子类中意义有明显差异，要么不 应继承，要么应该使用不同的基类

  1. 确保只继承需要继承的部分

  2. 不要覆盖一个不可覆盖的成员函数

    假设有一个函数在基类中是private的，那么可以在子类中创建一个同名的类，这样并没有覆盖只是刚好同名，但是看上去会是多态的，因此应该避免这种情况

  3. 只有一个派生类的基类是值得怀疑的（根本没有意义，不能精简代码反而还复杂化）

  4. 避免让继承体系过深，即不要太多层继承，这样很容易增加失误的概率，中间一个地方出错也很难找到错误处

  5. 尽量使用多态，避免类型检查

    有的时候会使用的这种代码

    此时可以直接使用多态，用一条语句就够了

成员函数

  1. 让类中的子程序尽可能地少，子程序越多越容易发生问题

  2. 减少成员函数之间的间接调用

    类似于以上的调用应该尽量避免

构造函数

  1. 尽可能在所有的构造函数中初始化所有成员

  2. 可以使用私有构造函数来强调单一实例

应当避免的类

  1. 避免创建万能类（解耦）

  2. 消除无关紧要的类

    如果一个类只包含数据而不包含行为，那么这种类没有存在的必要，此时应当将其降级，拆分为具体的数据或者成为多个类的属性

  3. 避免使用动词命名的类，一个由行为定义的类是没有意义的。类应当是一个客观实体

核对表

抽象

  1. 是否有一个中心目的
  2. 类的名称是否恰当
  3. 类是否展现一致的抽象
  4. 类的接口是否足够明了
  5. 类的接口是否足够抽象，可以将其视为黑盒
  6. 类提供的服务是否完整，不用使用私有数据
  7. 是否去除无关信息
  8. 是否可以分解为组件类

封装

  1. 是否将可访问性降至最低
  2. 是否会暴露数据成员
  3. 是否对其他类隐藏实现细节
  4. 是否避免对于使用者作假设
  5. 是松散耦合的吗

继承

  1. 派生类是否遵循Liskov替换法则
  2. 类的文档中应记录继承关系
  3. 派生类是否覆盖不可覆盖的方法
  4. 是否将公用的接口放到较高的继承层次
  5. 继承层次是否较浅
  6. 基类的所有数据成员是否定义为private而不是protected

实现

  1. 是否有七个或更少的数据成员
  2. 是否进行初始化

高质量的子程序

子程序是位实现一个特定目的而编写的程序，如函数或者方法

即便是一个很短的语句，将其使用函数编写能够增大程序的可读性

内聚性指的是子程序中各操作之间的联系紧密程度，内聚性越高越好

如sin()这个求正弦函数的方法就是极端内聚的，而sinAndCos()同时处理正弦和余弦这种操作就是不那么内聚的。

最好要做到一个子程序有很高的功能上的内聚性

子程序的名字

  1. 应当描述所做的事情

  2. 避免使用模糊的动词如handleService这种在大部分时候都是指代不明的

  3. 最好不要通过数字来区分不同的子程序

  4. 长度应当在9-15个字母之间

  5. 给过程起名的时候最好使用的语气强烈的动词加上名词

  6. 为常用操作确定统一的命名方式

    以上各种方式应当选择一种，以便之后不容易忘记，否则类多了之后会发生混淆

子程序的长度

子程序最好不要超过200行代码，如果超过了500行代码就应该考虑将其拆分为多个子程序了

子程序的参数

特别注意不要将参数作为工作变量

在程序较短的时候还好，如果程序较长的话很容易弄混

防御式编程

防御式编程即保证即便在其他程序员作出错误的操作的时候，仍然能够正确地处理，并返回错误信息而不是直接报错乃至程序崩溃

断言

断言常用于表示程序在执行时发生的意料之外的情况

如假定传入500个数据，那么就可以使用断言，如果传入数据超过了500个就会发生报错

断言一般用于以下情况

  1. 输入参数的取值范围
  2. 输入输出流的开关状态
  3. 文件的访问性
  4. 是否更改不应更改的变量
  5. 指针非空

断言一般只用于开发和测试阶段，是不会展现给用户的

使用断言应当遵循以下的建议：

  1. 用错误/异常处理预期会发生的事，使用断言来处理绝对不会发生的事
  2. 不应把需要执行的代码放到断言中
  3. 用断言来验证前后条件（如输入的城市应当是已经有的）
  4. 可以断言和错误处理同时使用
