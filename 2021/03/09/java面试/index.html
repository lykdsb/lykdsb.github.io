JVM相关

java的垃圾回收机制

java程序员不需要考虑内存的管理，垃圾回收器通常作为一个单独的低级别线程运行，不可预知的情况下对内存堆中的已经死亡或者很长时间没有使用的对象进行清除和回收

程序员不能实时控制垃圾回收器的运行

程序员可以使用System.gc()通知GC的运行，但是java并不保证GC的执行

JVM的内存模型

  • 程序计数器：当前线程执行字节码的行号指示器，是独立存储的，是线程私有的（即每一个线程都有独立的程序计数器，多个程序计数器独立存储互不影响）

  • 虚拟机栈：每一个方法执行的时候都有一个栈帧，用于存储变量表、出口等（一般简称为栈）

    • 如果线程请求的栈深度大于虚拟机所允许的深度，将会抛出StackOverFlowError
    • 如果可以动态扩展的话，如果无法申请到足够的内存的话就会抛出OutOfMemoryError

  • 本地方法栈：虚拟机栈是为执行java方法服务的，而本地方法栈是为虚拟机用到的Native方法服务的

  • 堆：用于存储对象实例，所有的对象实例都在这里分配内存。堆也是垃圾收集器的主要工作区

  • 方法区：用于存储已经被虚拟机加载的类信息等数据。常量池是方法区的一部分，用于存放各种字面常量和符号引用

    通过反射获取到的类型、方法名、字段名称、访问修饰符等信息就是从方法区获取到的。

JVM的四种引用

  • 强引用 ： 对象如果具有强引用则垃圾回收器绝对不会进行回收

  • 软引用：只要内存够用就不会进行回收，如果内存不足就会回收这些对象 的内存

  • 弱引用：无论内存是否足够，只要JVM开始进行垃圾回收，那么被弱引用关联的对象都会被回收

  • 虚引用：和没有引用没有差别，随时可能被GC回收

对象创建的方法

  1. 使用new进行对象的创建

  2. 调用对象的clone方法

    同时满足以下两个条件

    1. 被克隆的类需要实现Cloneable接口

    2. 重写clone方法

      • 深拷贝：对于值类型的成员变量进行值的复制，对于引用类型的成员变量也进行引用对象的复制
      • 浅拷贝：对于值类型的成员变量进行值的复制，对于引用类型的成员变量只复制引用，不复制引用的对象

  3. 使用反射，Class.forName().getInstance()

  4. 运用反序列化的手段

内存泄露和内存溢出

内存泄露指的是一个不再被程序使用的对象仍然在内存中占有存储空间

内存泄露最典型的例子是：没有重写hashcode和equals方法的Key类在HashMap中保存后生成很多重复的对象

防止内存泄漏的方法

  1. 避免在循环中创建对象
  2. 尽早释放无用对象的引用
  3. 尽量少用静态变量（其放在方法区。基本上不参与垃圾回收）
  4. 字符串处理，避免使用String，尽量使用StringBuffer，每一个String对象都得独立占用内存中的一块的区域
  • String的值是不可变的，每次对于String的操作都会产生新的对象，效率很低(适用于少量数据操作)
  • StringBuffer是可变类，对于其进行的操作不会产生新的对象。有一定的缓冲区容量，字符串大小没有超过的时候不会进行分配容量（适用于多线程的大量数据操作）
  • StringBuilder是可变类，进行操作的速度是很快的，但是线程不安全（适用于单线程的大量数据操作）

如果内存存在持续的增长就说明存在内存泄露

内存溢出指的是程序运行过程中没有办法申请到足够的内存而产生的错误

导致内存溢出的原因：

  • 内存中加载的数据量过于庞大
  • 集合类中有对对象的引用
  • 代码中存在死循环
  • 启动参数内存值设置的太小了

解决方法：

  1. 修改JVM的启动参数，增加内存
  2. 检查错误日志，看发生错误的类型
  3. 进行代码检查
  4. 查看内存使用情况

减少GC发生的次数

  1. 对象不用时显式设置为NULL
  2. 尽量少使用system.gc()
  3. 尽量少使用静态变量，其属于全局变量，不会被回收
  4. 尽量使用StringBuffer
  5. 分散对象创建或删除的时间
  6. 尽量少使用finalize函数
  7. 尽量使用基本数据类型，int而不是Integer

java编译过程

  1. 编译：将文件编译为.class文件。编译过程中优先编译依赖的类

    字节码文件格式主要有两个部分

    • 常量池
    • 方法字节码

  2. 执行：输入字节码文件，字节码解析，输出执行完的结果

栈溢出的原因和解决方法

  1. 大量的递归调用导致超过栈容量
  2. 分配了大量的局部变量

解决方式：

  1. 将递归改变为非递归
  2. 使用静态对象替代非静态局部对象
  3. 增加堆栈的大小

Hashmap的加载因子

加载因子是Hash中元素的填满程度

如果加载因子越大，则空间利用率增加，冲突的机会增加

如果加载因子越小，则不容易产生冲突，但是空间利用率低

HashMap中的key

HashMap中的key可以是任何对象，不管是可变对象还是不可变对象

可变对象作为key的时候需要重写equals方法和hashCode方法，保证对象属性改变的时候hashcode的返回值不发生变化

java的跨平台

Java的跨平台是虚拟机的跨平台，java代码编译之后产生class文件，java虚拟机就是将class文件翻译为特定平台的机器吗再进行运行

java的泛型

java生成的字节码是不包含任何范型信息的，范型信息在编译的过程中将会被擦除，将泛型转换为普通的java代码

面向对象的六个基本原则

  1. S 单一职责原则：类的功能要单一，一个类只负责一个职责
  2. O 开放封闭原则：软件应该对于扩展开放，对于修改封闭
  3. L 里氏替换法则：任何能够使用基类的地方都能够使用子类进行替换
  4. I 接口隔离：将接口的粒度最小化，将接口达到高内聚
  5. D 依赖倒置： 类药依赖于抽象而不是具体，即面向接口编程

对于小于int的数据类型进行运算的时候，会将其转换为int类型，因此次如果将两个short类型进行相加之后得到的结果是int类型

java程序的初始化顺序

  1. 父类的静态变量、父类的静态代码块 (谁在前,谁先初始化)
  2. 子类的静态变量、子类的静态代码块(谁在前,谁先初始化)
  3. 父类的非静态变量、父类的非静态代码块(谁在前,谁先初始化)、父类的构造函数  
  4. 子类的非静态变量、子类的非静态代码块(谁在前,谁先初始化)、子类的构造函数

构造函数

java中的构造函数是不能够进行继承的，子类通过调用super来显式调用父类的构造函数

异常

Throwable是所有异常的父类，该类有两个子类Error和Exception

Error：非常严重的错误，是不能够恢复的

Exception：是可以恢复的异常，包含两种类型

  • 检查类型（FileNotFoundException）
  • 运行时异常（RuntimeException）

java面向对象的三个特征与含义

  • 封装

  • 继承

重载和重写

重载：是在一个类中多态性的一种表现, 是指在一个类中定义了多个同名的方法, 但是他们有不同的参数个数或有不同的参数类型。

重写：子类对于父类方法的覆盖

final用来修饰类、变量、方法，表示其不能进行修改和扩展

finally是异常处理的时候一定会执行的代码

finalize是垃圾回收器工作的时候调用的方法

java的优点

  1. 跨平台
  2. 垃圾回收机制
  3. 去掉了容易引起错误的指针和运算符重载
  4. 有比较好的安全性（数组边界检测）

接口与抽象类的区别

  1. 语法层面
    1. 抽象类可以提供成员方法的实现细节（只是不能实例化），但是接口中只能提供抽象方法
    2. 抽象类中的成员变量可以是各种类型的，但是接口中的成员变量只能是public static final的
    3. 抽象类可以有静态代码块、静态方法和构造方法，但是接口中不能
    4. 一个类只能继承一个抽象类，但是可以实现多个接口
  2. 设计层面
    1. 抽象层次的不同：抽象类是对于类的抽象，但是接口只是对行为的抽象
    2. 跨域不同
    3. 抽象层次不同：抽象类在设计的时候是自下而上的，先知道子类才能设计出父类。而接口的设计不需要知道实现类的存在

内部类

  1. 静态内部类

    静态内部类只能访问外部类的静态成员和静态方法

  2. 非静态内部类

    能够访问外部的非静态成员

    静态内部类不依赖于外部类的实例化而实例化，但是非静态内部类需要在外部类实例化之后才能够实例化

  3. 局部内部类

  4. 匿名内部类

    • 匿名内部类不能有构造函数
    • 只能创造匿名内部类的一个实例
    • 匿名内部类一定是在new的后面，必须继承一个父类或者实现一个接口

java中的集合类

ArrayList、LinkedList、Vector

ArrayList、vector都只能按顺序进行元素的存储，删除的时候需要移位并且置空，这两种事基于数组实现的

而LinkedList是基于双向循环链表实现的

  • 安全性
    • ArrayList、LinkedList是线程不安全的
    • Vector是线程安全的，大部分的方法都含有关键字synchronized，但是其效率比较低

HashMap、HashTable

HashMap的存储函数实现

  1. 获取key值
  2. 调用key中重写的hashcode函数
  3. 对于该哈希值进行再哈希（使得分散更加均匀）
  4. 将哈希值和数组长度-1进行按位与的运算，（由于数组长度是2的幂），因此作用时将哈希值的高位去除，获取较低位分布在多个链表中。这样得到数组下标
  5. 将值存储到该位的链表中（进行一系列的判断）
  • 数组长度是2的整数次幂是因为：这样进行与操作的速度会非常高
  • 当length为2的整数次幂的时候，h&(length-1)等价于h%length
  • &具有比%更高的效率

当节点个数超过数组大小的加载因子时，会进行数组扩容

HashTable与HashMap的机制基本上一致

但是HashTable时线程安全的

HashTable中不允许null值，但是HashMap允许

因此判断是否存在某个键的时候，HashMap不能使用get方法进行判断，应该使用containsKey方法进行判断。但是HashTable可以直接使用get()==null进行判断
